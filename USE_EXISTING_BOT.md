# 使用现有飞书机器人

如果你已经有一个飞书机器人，可以直接使用它来访问知识库，无需创建新应用。

## ✅ 前置条件检查

### 1. 确认机器人类型

**可以使用：**
- ✅ 企业自建机器人
- ✅ 自己创建的机器人

**不能使用：**
- ❌ 从应用市场安装的机器人（权限受限）
- ❌ 他人分享的机器人

### 2. 确认机器人权限

需要检查机器人是否有以下权限：
- `wiki:wiki:readonly` - 获取知识空间信息
- `docx:document:readonly` - 查看文档

---

## 🔧 配置步骤

### 步骤 1: 找到机器人的凭证

1. 访问 https://open.feishu.cn/
2. 登录后进入【开发者后台】
3. 找到你的机器人应用
4. 进入应用详情页

你会看到：
```
App ID: cli_xxxxxxxxxxxxxxxxx
App Secret: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### 步骤 2: 检查并添加权限

1. 在应用详情页，点击【权限管理】
2. 搜索并添加权限：

**必需权限：**
- `wiki:wiki:readonly` - 获取知识空间信息
- `docx:document:readonly` - 查看文档

**如果没有这些权限：**
1. 点击【添加权限】
2. 搜索 `wiki` 和 `docx`
3. 勾选上述权限
4. 点击【确定】

### 步骤 3: 重新发布（如果添加了新权限）

如果你添加了新权限：
1. 点击【版本管理与发布】
2. 创建新版本
3. 申请发布
4. 等待审批通过（如果你是管理员可以自己审批）

### 步骤 4: 配置到本地

编辑 `.env` 文件：

```bash
# 使用机器人的凭证
FEISHU_APP_ID=cli_xxxxxxxxxxxxxxxxx     # 机器人的 App ID
FEISHU_APP_SECRET=xxxxxxxxxxxxxxxx      # 机器人的 App Secret
FEISHU_WIKI_SPACE_ID=N9kgwANVwifcisk9UT6cDOFInRe
```

### 步骤 5: 测试连接

```bash
python test_api_access.py
```

## 🎯 使用场景对比

### 场景 1: 只用于离线检索

**推荐方案：创建新的自建应用**

原因：
- 权限更清晰
- 不影响机器人的其他功能
- 更好的权限隔离

### 场景 2: 机器人本身要集成检索功能

**推荐方案：直接使用机器人凭证**

示例：
```python
# 用户在飞书中 @机器人 提问
用户: @检索助手 产品如何使用？

# 机器人后端：
1. 接收消息
2. 使用本 Skill 检索知识库
3. 返回检索结果给用户
```

这种情况下，使用同一个机器人的凭证更合理。

### 场景 3: 已有机器人，想快速测试

**推荐方案：先用机器人测试，后续可换成独立应用**

步骤：
1. 用机器人凭证快速测试
2. 验证可行性
3. 如需长期使用，创建独立应用

## ⚠️ 常见问题

### Q1: 我的机器人能访问知识库吗？

**检查方法：**
1. 进入机器人的【权限管理】页面
2. 查看是否有 `wiki:wiki:readonly` 权限
3. 如果没有，点击【添加权限】

### Q2: 添加权限后还是无法访问？

**可能原因：**
1. 权限未生效（等待 1-2 分钟）
2. 应用未重新发布
3. Space ID 不正确

**解决：**
```bash
# 运行测试脚本查看详细错误
python test_api_access.py
```

### Q3: 机器人和独立应用有什么区别？

**功能上：** 没有区别，都是调用同样的 API

**管理上：**
- 机器人：可能混合多种功能和权限
- 独立应用：权限更清晰，职责单一

**建议：**
- 测试阶段：用机器人快速验证
- 生产使用：创建独立应用

### Q4: 使用机器人会影响它的聊天功能吗？

**不会！** 这只是复用机器人的 API 凭证，不影响其消息功能。

### Q5: 能同时用于聊天和检索吗？

**可以！** 这是很好的使用场景：

```python
# 机器人架构
飞书机器人
├── 消息处理（原有功能）
│   ├── 接收用户消息
│   └── 回复用户
└── 知识库检索（新增功能）
    ├── 使用本 Skill 检索
    └── 返回结果
```

## 🔄 从机器人迁移到独立应用

如果你先用机器人测试，后续想换成独立应用：

### 迁移步骤

1. **创建新应用**
   - 参考 [SETUP_GUIDE.md](SETUP_GUIDE.md)

2. **配置相同权限**
   - `wiki:wiki:readonly`
   - `docx:document:readonly`

3. **更新 .env 文件**
   ```bash
   # 替换为新应用的凭证
   FEISHU_APP_ID=新的App ID
   FEISHU_APP_SECRET=新的App Secret
   ```

4. **测试**
   ```bash
   python test_api_access.py
   ```

5. **无需重建索引**
   - 数据在本地，凭证切换不影响

## 💡 最佳实践建议

### 方案 A: 分离使用（推荐）

```
机器人应用
└── 用途：消息交互、聊天对话

独立应用（本 Skill）
└── 用途：知识库检索、离线索引
```

**优点：**
- 职责清晰
- 权限分离
- 便于管理

### 方案 B: 复用机器人

```
机器人应用
├── 消息交互
└── 知识库检索
```

**优点：**
- 配置简单
- 统一管理

**缺点：**
- 权限混杂
- 难以追踪 API 调用

## 🎯 快速决策

### 选择机器人凭证，如果：
- ✅ 你已经有机器人
- ✅ 机器人有知识库权限
- ✅ 只是快速测试
- ✅ 机器人本身要集成检索功能

### 创建独立应用，如果：
- ✅ 长期使用
- ✅ 希望权限分离
- ✅ 便于管理和监控
- ✅ 多人协作开发
